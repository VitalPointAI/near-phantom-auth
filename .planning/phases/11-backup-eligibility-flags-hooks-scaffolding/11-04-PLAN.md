## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `passkeyManager.finishRegistration` (in-process) → `/register/finish` HTTP response | `passkeyData` returned by the manager is the trusted source for the new BE/BS bits — derived from `@simplewebauthn/server` verification. |
| HTTP response body → consumer (browser) | Library promises additive-only contract; new `passkey` key is opt-in. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-11-04-01 | Information Disclosure | New `passkey` field on `/register/finish` (anonymity invariant) | mitigate | Field is two booleans only — `{ backedUp: boolean; backupEligible: boolean }`. The shape is enforced by the type contract from 11-01. No `codename`, `userId`, `nearAccountId`, `email`, `ip`, `userAgent`, or any other identifying field is added. (RESEARCH.md ## Security Domain row "Anonymity invariant breach".) |
| T-11-04-02 | Tampering | Response shape regression — existing top-level fields reordered or replaced | mitigate | Pattern S4 ("Additive response shape — never reorder, never replace") followed: new `passkey` key is APPENDED to the literal. Acceptance criteria assert that `success`, `codename`, `nearAccountId` are all preserved (their existing greps continue to count ≥1). Tests use `expect.objectContaining` / `toMatchObject` rather than full deep-equal so a future field addition does not silently break consumers. |
| T-11-04-03 | Tampering | Stale `passkey.backedUp` value (Pitfall 1 — DB row used as source) | mitigate (does not apply at register, but enforced anyway) | Register-finish reads `passkeyData.backedUp` directly from `finishRegistration` — that value is freshly parsed from the assertion's authData by `@simplewebauthn/server`, never from the DB. The same data source is used as for the existing `createPasskey` call that writes `backed_up` to the DB. (RESEARCH.md Pitfall 1 — the risk is on /login/finish, addressed in 11-05; this plan applies the same pattern proactively.) |
| T-11-04-04 | Tampering | Helper input shape drift (Pitfall 2 — caller passes wrong type) | mitigate | The call site is `deriveBackupEligibility(passkeyData.deviceType)` — `passkeyData.deviceType` is typed as `'singleDevice' \| 'multiDevice'` by `passkey.ts:finishRegistration` (which inherits the type from `@simplewebauthn/server@13.2.3`). tsc rejects any wider input. |
