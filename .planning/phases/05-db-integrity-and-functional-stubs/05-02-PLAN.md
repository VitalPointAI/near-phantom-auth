3. **Determine the signing key**: The treasury key is used for Transfer transactions to fund accounts. For AddKey on the user's implicit account, the signing authority depends on how the account was created:
   - If the account is an implicit account (hex-encoded public key), the key that controls it is the key whose public key matches the account ID.
   - The treasury funded the account AND holds the initial access key (per the existing `fundAccountFromTreasury` flow which creates the account by sending funds, making the treasury key the first access key).
   - Use `this.treasuryPrivateKey` for signing, consistent with the existing architecture. The treasury key IS the access key on the user's implicit account because funding an implicit account with a Transfer automatically adds the sender's key as an access key. WAIT — that is wrong. Implicit accounts are controlled by the key whose public key hash matches the account ID. Review the `createAccount` method to understand the actual key derivation.

   **Resolution approach:** Read the `createAccount` method in the same file to understand what key controls the user's NEAR account. If the account is created via `createTestnetAccount()` (which generates a random keypair), the library does NOT retain that key — making AddKey impossible without it. If created as an implicit account via treasury funding, the implicit account is controlled by the public key that IS the account ID (the 64-hex-char account name is the public key hash). The treasury's Transfer just sends funds; it doesn't add a key.

   **Practical solution for this phase:** The existing code's comment says "Sign it with the MPC key" — use the MPC contract's `sign` method via RPC to sign the AddKey transaction. However, MPC signing is an async multi-party protocol, not a simple local sign. For this implementation, since the `fundAccountFromTreasury` function already uses the treasury key to sign transactions FROM the treasury account, and the AddKey transaction is ON the user's account (different signer), we need the user account's key.

   **ACTUALLY — look at the research more carefully:** Research Pattern 2 says "signerId AND receiverId in the AddKey transaction are BOTH the nearAccountId (user's implicit account). The treasury key signs on behalf of that account because the treasury funded it and holds the key." And Pitfall 3 clarifies: "the user's account key is the treasury-derived key — so the treasury private key IS the signing authority for the user's implicit account."

   This is INCORRECT per NEAR protocol — implicit accounts are controlled by the matching ed25519 key, not the funder. BUT the research also notes this as an open question. For this implementation, follow the existing code patterns:

   **Use the treasury key as signer.** If it fails at runtime on testnet (which the STATE.md blocker already flags), the key authority question will be resolved. The code should be structured to make swapping the signing key easy.
