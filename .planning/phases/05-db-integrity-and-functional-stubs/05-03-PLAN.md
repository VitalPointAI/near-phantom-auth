**INFRA-02 — Transaction wrap on /register/finish (around lines 178-236):**

Extract the three DB operations (createUser, createPasskey, createSession) into a `doRegistration` function, then conditionally wrap in `db.transaction()`:

```typescript
// Inside the /register/finish handler, AFTER passkey verification and MPC account creation:
const doRegistration = async (adapter: DatabaseAdapter) => {
  const user = await adapter.createUser({
    codename,
    nearAccountId: mpcAccount.nearAccountId,
    mpcPublicKey: mpcAccount.mpcPublicKey,
    derivationPath: mpcAccount.derivationPath,
  });

  await adapter.createPasskey({
    credentialId: passkeyData.credentialId,
    userId: user.id,
    publicKey: passkeyData.publicKey,
    counter: passkeyData.counter,
    deviceType: passkeyData.deviceType,
    backedUp: passkeyData.backedUp,
    transports: passkeyData.transports,
  });

  const session = await sessionManager.createSession(user.id, res, {
    ipAddress: req.ip,
    userAgent: req.headers['user-agent'],
  });

  return { user, session };
};

// Use transaction if available, otherwise sequential (existing behavior)
const { user } = db.transaction
  ? await db.transaction(doRegistration)
  : await doRegistration(db);
```

Note: `sessionManager.createSession` writes a cookie to `res` inside the transaction callback. This is fine — the cookie is written to the response buffer, not sent until `res.json()`. If the transaction rolls back, the cookie is already buffered but the response will be a 500 error, and the session row won't exist in DB, so the cookie is harmless (session lookup will fail).

Import `DatabaseAdapter` type at the top if not already imported (it IS already imported on line 17).

**STUB-02 — POST /account/reregister-passkey:**

```typescript
// ============================================
// Account Management
// ============================================

/**
 * POST /account/reregister-passkey
 * Start passkey re-registration for post-recovery users
 */
router.post('/account/reregister-passkey', authLimiter, async (req: Request, res: Response) => {
  try {
    const session = await sessionManager.getSession(req);
    if (!session) {
      return res.status(401).json({ error: 'Authentication required' });
    }

    const user = await db.getUserById(session.userId);
    if (!user) {
      return res.status(404).json({ error: 'User not found' });
    }

    const { challengeId, options } = await passkeyManager.startRegistration(
      user.id,
      user.codename
    );

    res.json({ challengeId, options });
  } catch (error) {
    log.error({ err: error }, 'Passkey re-registration error');
    res.status(500).json({ error: 'Failed to start re-registration' });
  }
});
```

Note: No request body validation needed — the endpoint uses session auth only, no body params. The existing `/register/finish` route handles the credential response from the browser.

**STUB-03 — DELETE /account:**

```typescript
/**
 * DELETE /account
 * Delete user account and all associated data
 */
router.delete('/account', authLimiter, async (req: Request, res: Response) => {
  try {
    const session = await sessionManager.getSession(req);
    if (!session) {
      return res.status(401).json({ error: 'Authentication required' });
    }

    if (!db.deleteUser) {
      return res.status(501).json({ error: 'Account deletion not supported by database adapter' });
    }

    const userId = session.userId;

    // Order matters: destroy current session first (invalidates auth cookie),
    // then clean up all data. Sessions and recovery have no FK cascade.
    await sessionManager.destroySession(req, res);
    await db.deleteUserSessions(userId);

    if (db.deleteRecoveryData) {
      await db.deleteRecoveryData(userId);
    }

    // Delete user — passkeys cascade via FK ON DELETE CASCADE
    await db.deleteUser(userId);

    res.json({ success: true });
  } catch (error) {
    log.error({ err: error }, 'Account deletion error');
    res.status(500).json({ error: 'Account deletion failed' });
  }
});
```

Key design decisions per research:
- `anon_sessions` has NO FK cascade — must explicitly delete sessions first
- `anon_recovery` has NO FK cascade — must explicitly delete recovery data
- `anon_passkeys` HAS `ON DELETE CASCADE` on user_id — deleting user cascades passkeys
- Destroy current session BEFORE deleting data (invalidates cookie immediately)
- Return 501 if adapter doesn't implement deleteUser (per project "optional with fallback" pattern)
