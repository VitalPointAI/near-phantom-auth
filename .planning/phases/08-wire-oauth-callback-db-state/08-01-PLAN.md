Two surgical edits to src/server/oauth/router.ts:

**Edit 1 — Move cookieParser() above the CSRF conditional (around line 68):**

Before the `if (config.csrf)` block, add:
```typescript
router.use(cookieParser());
```

Then REMOVE the `router.use(cookieParser())` call that is currently on line 88 inside the `if (config.csrf)` block. The `doubleCsrfProtection` middleware still works because cookieParser is already mounted above. Do NOT leave cookieParser mounted in both places.

The result should be:
```typescript
// cookieParser mounted unconditionally — OAuth callback needs req.cookies
// regardless of CSRF config; CSRF middleware also depends on it
router.use(cookieParser());

if (config.csrf) {
  const { doubleCsrfProtection } = doubleCsrf({ ... });
  // cookieParser already mounted above
  router.use(doubleCsrfProtection);
  log.info('CSRF protection enabled for OAuth router (callback exempt)');
}
```

**Edit 2 — Replace cookie comparison with oauthManager.validateState() in the callback handler (lines 198-209):**

REMOVE these lines:
```typescript
const storedState = req.cookies?.oauth_state;
if (state !== storedState) {
  return res.status(400).json({ error: 'Invalid state' });
}
const codeVerifier = req.cookies?.oauth_code_verifier;
res.clearCookie('oauth_state');
res.clearCookie('oauth_code_verifier');
```

REPLACE WITH:
```typescript
// DB-backed state validation — atomic lookup + delete (replay protection)
const oauthState = await oauthManager.validateState(state);
if (!oauthState) {
  return res.status(400).json({ error: 'Invalid state' });
}
const codeVerifier = oauthState.codeVerifier;
// Clear cookies for hygiene — they are no longer consulted for validation
res.clearCookie('oauth_state');
res.clearCookie('oauth_code_verifier');
```

Key points:
- `codeVerifier` MUST come from `oauthState.codeVerifier`, NOT from `req.cookies?.oauth_code_verifier`
- Keep `clearCookie` calls for hygiene (clean up browser cookies set during /start)
- The INFRA-05 guard (`if (req.cookies === undefined)`) can remain — it still fires if a consumer mounts the oauth router without the library's own cookieParser (e.g., sub-app isolation). However, with unconditional cookieParser mounting, the guard will no longer fire in the standard path. It stays as defense-in-depth.
- Do NOT change the `/start` handler — it still sets cookies (harmless, and provides fallback context)
