    ## Lazy-Backfill Hook (v0.7.0)

    `near-phantom-auth@0.7.0` exposes `hooks.backfillKeyBundle` — a **pass-through**
    hook that fires inside `POST /login/finish` when the authenticator returned a
    fresh PRF sealing key (`sealingKeyHex`). The hook lets consumers run their
    own key-bundle migration ceremony for pre-v0.6.0 NULL-bundle accounts —
    without the library taking any opinion on the consumer's schema, transaction
    boundaries, or recovery topology.

    The contract is "library steps aside": the library hands the consumer the
    minimal context (user identifiers + sealingKeyHex), invokes the hook, echoes
    the result on the response, and gets out of the way. **Backfill failure
    NEVER blocks login** (BACKFILL-03) — a hook throw is caught, the error is
    logged WARN with a redacted payload, and the response continues with
    `backfill: { backfilled: false, reason: 'skipped' }`.

    ### What the library does

    The library invokes the hook at exactly **one** fire point:

    - `POST /login/finish` — after passkey verify + `db.getUserById`, after any
      Phase 14 `hooks.afterAuthSuccess` that returned `continue: true`, BEFORE
      `sessionManager.createSession`. Fires **ONLY** when `sealingKeyHex` was
      supplied in the request body (no PRF -> no fresh sealing key -> silent
      skip; the hook is NOT invoked, no `backfill` field appears on the
      response).

    Sequential ordering with the Phase 14 second-factor hook (when both are
    configured):

    1. `hooks.afterAuthSuccess` runs first. If it returns `{ continue: false, ... }`,
       the response short-circuits with `secondFactor` echo and the backfill hook
       is NOT invoked.
    2. If `afterAuthSuccess` returned `{ continue: true }` (or was not configured),
       AND `sealingKeyHex` is in the request body, AND `hooks.backfillKeyBundle`
       is configured, the backfill hook fires.
    3. The result is echoed on the response under an additive `backfill` key,
       alongside the existing `success`, `codename`, and `passkey?` fields.

    The hook receives a single-shape context (no discriminated union — only one
    fire site):

    ```typescript
    import type { Request } from 'express';

    export interface BackfillKeyBundleCtx {
      userId: string;
      codename: string;
      nearAccountId: string;
      /** 64-char lowercase hex (validated upstream by loginFinishBodySchema).
       *  Sensitive material — consumer's hook owns its handling, library
       *  never logs this field. */
      sealingKeyHex: string;
      req: Request;
    }
    ```

    The hook returns:

    ```typescript
    export type BackfillReason =
      | 'already-current'   // consumer already has a current key bundle
      | 'no-legacy-data'    // no pre-v0.6.0 NULL-bundle row for this user
      | 'completed'         // consumer ran the backfill ceremony
      | 'skipped';          // consumer chose to skip (or library set this on hook throw)

    export interface BackfillKeyBundleResult {
      backfilled: boolean;
      reason?: BackfillReason;
    }
    ```

    Library echoes the hook result on the login response:

    ```jsonc
    {
      "success": true,
      "codename": "ALPHA-BRAVO-7",
      "passkey": { "backedUp": false, "backupEligible": false },
      "backfill": { "backfilled": true, "reason": "completed" }
    }
    ```

    On hook throw OR rejected Promise (BACKFILL-03 containment):

    - `sessionManager.createSession` IS still called (login completes 200 OK).
    - Response carries `backfill: { backfilled: false, reason: 'skipped' }`.
    - Library logs a single WARN entry with `redactErrorMessage(err)` payload —
      `Error.name` + first two stack frames; sealingKeyHex NEVER appears in the
      log (verified in CI by `src/__tests__/backfill-redaction.test.ts`).

    The library does NOT log or telemetrize any ctx field (`userId`, `codename`,
    `nearAccountId`, `sealingKeyHex`, `req`) — those are surfaced to the
    consumer's hook by design, but the library's pino emissions stay
    anonymous.

    ### What the consumer must do

    Wire the hook on `createAnonAuth({ ..., hooks: { backfillKeyBundle: ... } })`.
    The consumer fully owns the migration logic: read the legacy row, derive the
    new bundle from `ctx.sealingKeyHex`, persist atomically in the consumer's
    own transaction.

    Copy-pasteable example:

    ```typescript
    import { createAnonAuth } from '@vitalpoint/near-phantom-auth/server';
    import type { BackfillKeyBundleCtx, BackfillKeyBundleResult } from '@vitalpoint/near-phantom-auth/server';

    const auth = createAnonAuth({
      // ... db, mpcManager, etc.
      hooks: {
        async backfillKeyBundle(ctx: BackfillKeyBundleCtx): Promise<BackfillKeyBundleResult> {
          // 1. Look up the legacy NULL-bundle row in the CONSUMER'S schema.
          const legacy = await consumerDb.findLegacyKeyBundle({ userId: ctx.userId });
          if (!legacy) return { backfilled: false, reason: 'no-legacy-data' };
          if (legacy.bundle !== null) return { backfilled: false, reason: 'already-current' };

          // 2. Derive the new bundle envelope from ctx.sealingKeyHex.
          //    (Consumer's responsibility — library has no opinion on the envelope.)
          const newBundle = await deriveBundleFromSealingKey(ctx.sealingKeyHex, legacy);

          // 3. Persist in the CONSUMER'S transaction. Library does NOT wrap this.
          await consumerDb.transaction(async (tx) => {
            await tx.updateKeyBundle({ userId: ctx.userId, bundle: newBundle });
          });

          return { backfilled: true, reason: 'completed' };
        },
      },
    });
    ```

    `ctx.req` is the **bare Express Request** — it carries cookies, headers,
    body, etc. The library does NOT sanitize this surface; what your hook
    reads from `req` is your responsibility.

    ### Consumer-owns-schema contract (BACKFILL-04)

    The library is **explicitly pass-through**:

    - **Library does NOT persist key bundles.** There is no library-managed
      schema column, no library DB write, no library state mutation. The
      consumer brings their own table, their own column, their own write.
    - **Library does NOT run a transaction around the hook.** A hook throw
      is caught and the response continues; if the consumer needs atomicity
      across multiple writes, the consumer wraps their own writes in their
      own `consumerDb.transaction(...)`.
    - **Library does NOT enforce idempotency.** Login retries will invoke the
      hook on every successful authentication where `sealingKeyHex` was
      supplied. The consumer's hook MUST be idempotent — return
      `{ backfilled: false, reason: 'already-current' }` on second invocation.
    - **Library does NOT validate the consumer's schema.** The consumer's
      ceremony might fail mid-write; the library has no insight. Consumer
      observability (DB-side instrumentation, error rates) is consumer's job.

    ### Dual-recovery + IPFS-orphan footnote (BACKFILL-04)

    The library ships an IPFS-based recovery flow (`recovery/ipfs.ts`) that
    persists encrypted recovery blobs to IPFS keyed by user. Pre-v0.6.0
    NULL-bundle accounts use this flow as their primary recovery method.

    **The library does NOT migrate existing IPFS recovery blobs.** When the
    consumer's `backfillKeyBundle` ceremony replaces the recovery method
    (e.g., switches the user from IPFS to a key-bundle-based recovery),
    the IPFS blob remains in IPFS — referenced by no one. **The blob is
    orphaned.** It cannot be deleted from IPFS (content-addressed; no
    delete operation), and the library has no schema to track that the
    blob is now stale.

    This is the dual-recovery state: **the same user has both an IPFS
    recovery blob (still on IPFS) AND a fresh key bundle (in the consumer's
    schema), and only one of them is the active recovery path.** Which one
    the user lands on is determined by the recovery endpoint they hit
    (`/recovery/ipfs/recover` vs the consumer's own bundle-based recovery
    flow). Consumers MUST decide:

    - **Option A (consumer-managed deactivation):** the consumer maintains
      a flag in their schema (e.g., `recovery_method: 'ipfs' | 'bundle'`)
      and the consumer's recovery routes refuse to use the IPFS flow once
      the bundle is in place. The IPFS blob remains orphaned but the
      ambiguity is resolved at the consumer's gate.
    - **Option B (consumer-side cleanup):** the consumer's backfill
      ceremony explicitly invalidates the user's IPFS recovery blob (e.g.,
      by overwriting it with an "expired" marker, or by removing the
      consumer's reference to its CID from the user's row). The library
      offers no helper for this; consumers script it themselves.
    - **Option C (accept ambiguity):** small-deployment consumers may
      accept the dual-recovery state for the lifetime of v0.7.0 and plan
      to remove the IPFS flow entirely in a later milestone. The library
      does not warn against this; it is a valid posture for low-traffic
      deployments.

    The library does not pick an option for you. The trade-off is yours.

    ### Known limitation: hook timeout (T-15-02)

    The library awaits the hook with NO `Promise.race` timeout wrapper. A
    hook that hangs holds the login response open until the consumer's
    runtime (Express, Lambda, etc.) kills the request. **Consumer hooks
    MUST resolve in finite time.** Adding a library-side timeout is
    deferred to a future minor version (a timeout would change the
    semantics — does the library set `reason: 'skipped'` on timeout?
    Does it return a 5xx? — and requires its own contract negotiation).

    - REQUIREMENTS: `BACKFILL-01..04` in `.planning/REQUIREMENTS.md`
