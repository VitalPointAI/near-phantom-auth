## Integration Check Results

### Cross-Phase Wiring: 34/35 exports properly connected

**Resolved since previous audit:**
- INFRA-03: `oauthManager.validateState(state)` now called in callback (Phase 8, router.ts:203)
- INFRA-05 cookieParser: Now mounted unconditionally at router.ts:69, outside CSRF block (Phase 8)

**Remaining (LOW severity):**

**Dual oauthManager instances in createAnonAuth**
- `createAnonAuth` (index.ts:176) creates oauthManager instance A, exposed on `AnonAuthInstance.oauthManager`
- `createOAuthRouter` (oauth/router.ts:99) creates its own instance B internally
- For **postgres adapter**: Both share same DB — functionally benign (Phase 8 confirmed)
- For **custom adapters**: Both instances use the same closure in the router, so the in-memory fallback is self-consistent within the router. Instance A (`AnonAuthInstance.oauthManager`) is a dead export.
- Fix path: Add optional `oauthManager` param to `OAuthRouterConfig`, inject from `createAnonAuth`

**AnonAuthConfig.derivationSalt top-level field**
- Declared in types but `createAnonAuth` only forwards `config.mpc?.derivationSalt` (not top-level)
- Consumer must set `mpc.derivationSalt` — the top-level field is effectively dead
- SEC-04 is satisfied because the `mpc.derivationSalt` path works correctly
